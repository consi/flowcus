# Merge System Architecture
**Context:** How background compaction works in Flowcus storage
**Decision/Pattern:** Single coordinator (async task) + semaphore-gated blocking workers. Coordinator scans hour directories, builds merge plans, dispatches to tokio::spawn_blocking. Workers read column files from source parts, decode transforms (delta/gcd reverse), concatenate into combined ColumnBuffer, re-encode with auto-codec, write new part at generation+1, then delete source parts. Race prevention: HashSet<PathBuf> tracks active merges, coordinator is single decision-maker. Fair scheduling: reads /proc/loadavg and /proc/meminfo, scales effective worker count linearly below threshold. Past hours merge all parts aggressively (single part goal). Current hour merges in batches of 4 to stay responsive. Max generation = 65535, stop merging beyond that.
**Rationale:** Ingestion priority is preserved by throttling merge workers under load. Single coordinator prevents race conditions without distributed locks. Staging new part before deleting sources is crash-safe (worst case: duplicated data, not lost). Column-level decode+re-encode during merge allows re-evaluating codec choice on larger datasets (better compression).
